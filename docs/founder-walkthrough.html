Choose My Lawyer — what we've built so far

A walkthrough of the pilot: what a consumer experiences, how data flows, what the firm sees, and how ranking and pricing actually work under the hood.

The product in one line

A guided intake chat that turns a home-buyer's situation into a ranked, priced list of WMCA conveyancing solicitors — with one click to instruct or request a callback.

What's commercially novel

Ranking is deterministic, rules-based, and runs across the whole market — enrolled firms aren't promoted into the ranking, they're just the ones we can contact. That preserves the "independent comparison" position.

What the pilot is testing

Commercial validation, not feature completeness. Can ~90 days of PPC into the WMCA produce enough Proceed / Callback volume from a manually-priced firm panel to justify rollout?

2 · The pieces

Three runtime surfaces, one shared database, one in-process scheduler.

3 · The consumer journey

Click any step to see what gets written, sent, or calculated.

Land on marketing site → click "Find a solicitor"
Frontend  Meta Pixel fires PageView; mirrored to analytics_events.
The marketing pages live under frontend/app/(marketing). Pricing, "How it works", FAQ etc. All Pixel events are also POSTed to /api/public/events so we have an internal copy of every funnel hit.
Start the chat → POST /api/public/sessions
Backend  Creates a row in chat_sessions with a UUID; the URL becomes /chat/{uuid} so the session can be saved and resumed.
Sessions live in PostgreSQL (no Redis). Two JSONB columns: answers (the structured intake) and message_history (the rendered transcript). An expires_at column lets us garbage-collect abandoned sessions.
13-step intake (conveyancing)
Backend  Each answer → POST /api/public/sessions/{id}/answer · server validates per question type · stores into chat_sessions.answers JSONB.
The full schema lives in backend/app/services/chat.py. Sections:
  1. About your case — purchase price, freehold/leasehold, postcode, mortgage, new build, Help to Buy ISA, shared ownership.
  2. Ranking preference — one of 7 scorecard preferences; whether to include distance.
  3. Your details — first name, last name, email, phone (used for Proceed/Callback dispatch later).
The backend validates every answer (postcode regex, currency parse, email/phone shape) so a malformed payload can't get past the API.
Reveal results → GET /api/public/search/{session_id}
Backend  The ranker runs against the whole WMCA market, not just enrolled firms (see §7).
What happens server-side, in order:
  1. Load every non-intervened organisation, eager-loading offices, price cards, complaints decisions, regulatory decisions.
  2. Filter to firms with an active conveyancing price card AND a fee band that matches the purchase price (otherwise we can't quote them → exclude in pilot).
  3. Score each firm across 6 factors (reputation, price, complaints, regulatory, distance if opted-in, offices).
  4. Weight + rank by the user's chosen scorecard.
  5. Extract the top-5 contactable (enrolled) firms from the ranked list.
The result payload contains the full ranked market (so the user can scroll past the top 5) plus the top-5-contactable subset with rank/score/quote/factor breakdown.
Inspect a firm → "View breakdown"
Frontend  Modal shows itemised quote + factor scores + links to source LeO/SRA decisions.
Every results row carries the full quote breakdown (base fee + conditional adjustments + VAT + included disbursements) and the per-factor 0–100 scores. We also surface the source URLs of every complaints / regulatory decision so the user can see our working.
"Proceed" with a firm — POST /api/public/appointments (type=appoint)
Email  Three emails fire: to firm (in user's name, reply-to = user), copy to user, BCC to CML.
Server records:
  • An appointments row: type appoint, status pending, quoted price snapshot, conflict_check_outcome pending.
  • Consent flags (contacted + terms) captured from the form.
The firm email uses the user's name as from_name with their email as reply_to — the firm replies directly to the consumer, not to a CML relay. CML is BCC'd for audit.
Why this matters: regulators / firms prefer direct client contact; we just introduce.
"Request a callback" (up to 5 firms)
Email  Same dispatch pattern as Proceed; includes phone + preferred-time window. Type = callback.
A callback row carries client_name, client_email, client_phone, preferred_time (e.g. "Tuesday morning"). End-of-day a cron job emails the user asking whether the firm got in touch (see §6).
Follow-ups (scheduled, asynchronous)
APScheduler  Three daily cron jobs — see §6 for timings.
  • Callback follow-up — same working day, 17:00. Binary yes/no email; click writes appointments.firm_contact_made.
  • Proceed follow-up — 5 working days later. Quote-drift reminder + binary check-in.
  • Feedback request — 2 months later. Issues a review_invitations token; click goes to /review/{token}.
Conflict-check failure (rare)
Admin  Firm tells us privately they can't act; CML marks conflict_check_outcome=conflict; user emailed with a link back to their results.
Endpoint: POST /api/admin/appointments/{id}/conflict-check. The user can pick a different firm from their original ranked list — the session is still alive.
Review (post-matter)
Backend  Consumer opens token URL → submits rating + text → row in reviews, source=cml.
CML reviews are weighted 2× vs Google reviews when computing the cached aggregate_rating on the organisation. Firms can respond once per review and report inappropriate ones (the flagged ones queue for manual moderation; we don't have a moderation UI in MVP).

4 · The firm journey

Firms are invited manually for the pilot. No self-serve onboarding yet.

Admin issues an enrolment invitation
Admin  POST /api/admin/organisations/{org_id}/invite-enrollment → generates an enrollment_token UUID, emails the firm.
Organisation rows for every WMCA SRA-authorised firm already exist from the SRA CSV import. The invitation just attaches a token + sends a Sparkpost email containing the enrol link.
Firm clicks link → completes /enroll → creates account
Backend  POST /api/firm/auth/register: validates the token, hashes the password, marks enrolled=true and enrollment_token_used=true, creates a firm_users row.
We use JWT bearer tokens in localStorage (not HTTP-only cookies) — simpler for Next.js client components, acceptable for the firm portal which is non-public anyway.
Firm logs in → JWT issued
Backend  POST /api/firm/auth/login → returns access token. Frontend stores it; all /api/firm/* calls send it as Authorization: Bearer ....
A single FirmUser per organisation in MVP (admin role). Multi-seat is deferred.
Profile setup — /firm/profile
Frontend  PATCH /api/firm/profile — website, phone, public email. SRA number and name are read-only (sourced from the regulator).
The firm can also paste their Google Place ID, which the admin can then sync (see §6) to start pulling reviews.
Price card editor — /firm/pricing
Backend  POST/PUT /api/firm/pricing — stores the structured price card as JSONB on price_cards.
The editor captures fee bands by purchase price, conditional adjustments (e.g. leasehold supplement), VAT treatment, and included disbursements. The same JSON shape is used by the ranker's price calculator at runtime — see §9.

There's a "preview quote" endpoint that lets the firm see what their price card would output for a sample matter before they publish it.

Dashboard — /firm/dashboard
Backend  GET /api/firm/dashboard/stats — counts of Proceeds, Callbacks, average rating, etc.
No PII-rich lead list yet — appointments are private and the firm gets the lead by email directly (in the user's name). The dashboard exists to show the firm volume / pipeline at a glance.
Reviews management — /firm/reviews
Backend  GET /api/firm/reviews · POST /respond · POST /report.
Firms can reply once per review and flag obviously-bogus ones. Flagged reviews go into a queue but there's no in-app moderation tooling for the pilot — it's a human review by us.
Pilot constraint: firms are enrolled and priced by hand. We anticipate ~10–25 firms across WMCA at launch. Self-serve onboarding and automated pricing import are explicit non-goals for MVP.

5 · Admin / internal operations

There's no admin UI in MVP. Everything below is API-only — we run it from a tool like Postman / curl.

POST /api/admin/organisations/{id}/invite-enrollment

Generate enrolment token; email firm a join link.

POST /api/admin/organisations/{id}/sync-google-place

Manually pull Google reviews + place rating for an org (also runs weekly in cron).

POST /api/admin/appointments/{id}/conflict-check

Mark a Proceed as failed due to firm-side conflict; triggers the consumer "pick another solicitor" email.

GET /api/admin/analytics/export

CSV of every captured event (funnel, conversions). For founder-side analysis without depending on Meta.

6 · Scheduled jobs

All four jobs run in-process inside the FastAPI app via APScheduler. No separate worker.

JobCadenceWhat it doesWhy
Google reviews sync Mondays · 03:00 For every enrolled firm with a Google Place ID, fetch latest reviews + rating, upsert into reviews (source=google), recompute aggregate_rating. Reputation factor depends on this data being current.
Callback follow-up Daily · 17:00 For callbacks created today, email user "did the firm call you back?" with yes/no buttons. Holds firms accountable; populates firm_contact_made.
Proceed follow-up Daily · 09:00 For Proceeds made 5 working days ago, email user quote-drift reminder + binary check-in. Catches stalled instructions; price-staleness disclaimer.
Feedback request Daily · 09:30 For Proceeds made ~2 months ago, issue a review_invitations token and email the consumer. Source of CML-native reviews; replaces the old 90-day cron.
Failure mode to consider: APScheduler is in-process. If we run multiple backend instances, the same job runs once per instance. For pilot scale (one Railway service) this is fine. Post-pilot we'd extract to a single worker or use Railway cron.

7 · The six-factor ranking, plain English

Deterministic. Identical inputs always produce identical outputs. No runtime AI.

The six factors

  1. Reputation — rating × confidence weighting based on review count. Normalised across the result set.
  2. Price — Total Effective Price (fees + VAT + included disbursements). Normalised across the result set — cheapest gets 100.
  3. Complaints history — Legal Ombudsman decisions. Base 100, deductions per decision.
  4. Regulatory history — SRA + SDT decisions. Base 100, larger deductions per outcome.
  5. Distance — optional. Geodesic distance from user's postcode to nearest office. Normalised — closer wins.
  6. Number of offices — banded absolute score (1 office → 70 ... 21+ offices → 100).

Sequencing — the key trick

  1. Rank all in-scope WMCA firms (signed up or not).
  2. Extract the top-5 contactable (= enrolled) firms from that ranking.

We never rank enrolled firms separately. This preserves the "whole of market" positioning — the rankings are the same whether a firm pays or not; enrolment just unlocks the "contact" action.

Two binary eligibility rules

  • SRA Intervention on a firm → removed from results entirely (not a deduction — a hard exclusion).
  • No active conveyancing price card matching the purchase-price band → excluded from this pilot. Post-pilot, the "Estimated Price" path (normalised from public transparency statements) lifts this restriction.

Tie-break order

If two firms have the same overall weighted score:

reputation → complaints → regulatory → price → distance → offices → alphabetical

9 · How a quote is built

The price the user sees is computed at runtime from the firm's stored price card plus the user's answers — no fuzzy estimation.

Total Effective Price formula

P_legal_effective   = P_quoted                            (pilot: quoted only)
fees_subtotal       = base_fee + Σ matching adjustments
vat_on_fees         = fees_subtotal × 0.20                 (if VAT applies)
disbursements_total = Σ (each disbursement × its VAT flag)

P_effective         = fees_subtotal + vat_on_fees + disbursements_total

What sits in the price card (JSONB)

{
  "practice_area": "residential_conveyancing",
  "pricing_model": "band",
  "bands": [
    { "purchase_price_min": 0,      "purchase_price_max": 250000, "fee": 950 },
    { "purchase_price_min": 250001, "purchase_price_max": 500000, "fee": 1200 }
  ],
  "adjustments": [
    { "name": "Leasehold supplement",   "amount": 250, "condition": "tenure==leasehold" },
    { "name": "New build supplement",   "amount": 200, "condition": "new_build==true" },
    { "name": "Help to Buy ISA",        "amount": 50,  "condition": "help_to_buy_isa==true" }
  ],
  "vat_applies_to_fees": true,
  "included_disbursements": [
    { "name": "Land Registry fee",   "amount": 150, "vat_applies": false },
    { "name": "Bankruptcy search",   "amount": 6,   "vat_applies": true  }
  ]
}
Post-pilot: add an "Estimated Price" path — we normalise public firm transparency statements into the same structure ahead of time and apply a 7.5% confidence uplift at runtime. Disbursement estimates get a 2% uplift. Excluded disbursements (Stamp Duty etc.) are flagged to the consumer with a link to the CML explainer rather than baked in.

11 · How the CSVs become rankable data

Three ingest scripts, each idempotent. They populate the structured tables that the runtime ranker consumes.

scripts/import_sra_csv.py

Source: SRA "check a solicitor or firm" CSV download. The whole authorised firm register, filtered to our pilot region.

Columns we read

Organisation name SRA number Status Address line 1 / 2 City County Postcode Website Phone Email

What ends up in the DB

CSV fieldTarget column
Organisation name→organisations.name
SRA number→organisations.sra_number
Status→organisations.auth_status (filtered to "authorised")
Website / Phone / Email→organisations.website_url / phone / email
Address / City / County / Postcode→offices.* (primary office row)
Postcode → Fetchify (--geocode)→offices.lat / offices.lng

Run

python scripts/import_sra_csv.py --csv sra_firms.csv \
  --region "West Midlands" --geocode
Note: only firms with status = authorised survive the filter. Multi-office firms get a single primary office row from the CSV; additional offices are out of scope for ingestion right now.

scripts/import_leo_csv.py

Source: Legal Ombudsman published decisions. Firms are matched to SRA numbers before ingest (one-time entity resolution, done manually for the pilot).

Filter rules

  • provider_type must be "Firm SRA" or "ABS Firm SRA" (drops barristers / individual practitioners).
  • poor_service_found must be truthy.
  • sra_number must already exist in organisations (otherwise we drop the row).

Columns we read

decision_id sra_number decision_date provider_type poor_service_found remedy_type remedy_amount complaint_handling source_url

What happens at ingest (the key insight)

The script doesn't just dump rows — it pre-computes the severity_score, remedy_amount_score, and complaint_handling_penalty flag. The runtime ranker then just adds numbers.

CSV fieldHow it's mapped
remedy_type→severity_band (0 = no remedy ... 3 = financial compensation), then severity_score (0.30 / 0.50 / 0.60 / 1.00)
remedy_amount→remedy_amount_score via banding table (£0 → 10, ... £50k+ → 62)
complaint_handling→complaint_handling_penalty bool (adds +4 to deduction at runtime)
source_url→stored verbatim — shown to consumer in results modal

How the runtime uses this

per_decision_deduction = severity_score × remedy_amount_score
                       + (4 if complaint_handling_penalty else 0)

complaints_score       = 100 − Σ per_decision_deduction   (no floor)

Run

python scripts/import_leo_csv.py --csv leo_decisions.csv

scripts/import_sra_decisions_csv.py

Source: SRA-published regulatory decisions (rebukes, fines) AND Solicitors Disciplinary Tribunal fines. Same table; source column distinguishes them.

Special rule — Intervention

If decision_type = intervention, we don't insert a decision row. We flip organisations.intervened = true. That firm is then excluded from results entirely (binary eligibility, not a score deduction).

13 · Open questions / for discussion

Things to challenge, decide on, or scope before go-live.

Conflict-check workflow — should this be self-serve for firms?

Today: firm tells us privately a conflict exists; an admin runs POST /admin/appointments/{id}/conflict-check. Could (should?) firms mark this themselves from a firm-portal page? Trade-off is firms having to log in vs CML staff handling exceptions.

Billing — pilot is manual. What's our threshold for automating?

Chargeable events (Proceed, Callback) are recorded but never invoiced from-system. After ~N firms or ~£X/month do we automate? Need a decision on PSP (Stripe? Direct invoice via Xero?) and the dispute window after the chargeable event.

Review moderation — flagged reviews have no UI

Firms can flag reviews as inappropriate but there's no admin moderation screen. Pilot volume is probably zero, but if it's not, what does the workflow look like?

Multi-office firms — only one office gets imported from the SRA CSV

The "number of offices" factor reads org.offices count, which is currently 1 for everyone. We'll need a richer office source (or a manual add-office UI for enrolled firms) for the offices factor to actually do work.

Estimated Price path — deferred, but what's the trigger?

Currently a firm with no quoted price card is excluded from results. Post-pilot, we'd normalise public transparency statements offline and serve those with a confidence uplift. What's the proof point that justifies this work — number of WMCA firms missing from results? Time to enrol N firms?

Funnel telemetry — Meta Pixel + our DB mirror

Every Pixel event is written to analytics_events. Is the founder happy reading CSV exports, or do we need a basic dashboard before launch? GA4 is a possible addition; cost is one more cookie banner and one more vendor.

Pilot exit criteria — what would make us roll out vs iterate?

What's the minimum conversion rate / lead volume / firm satisfaction across the 90-day window for us to declare validation and progress to national rollout / additional practice areas?

Domain & brand at go-live

Cloudflare DNS will point to Railway; Sparkpost domain authentication needed for Proceed/Callback emails to land in firm inboxes. Worth confirming SPF/DKIM are set up well before the go-live date.

What's intentionally not built (per spec)

  • Practice areas other than residential conveyancing.
  • Geography outside WMCA.
  • Self-serve firm onboarding.
  • Probabilistic / AI pricing at runtime.
  • Case-management integrations.
  • Payments (Stripe / Xero / PSP).
  • Facebook / Trustpilot review aggregation.
  • In-app review moderation tools.
  • Voice input.
  • Welsh language.
